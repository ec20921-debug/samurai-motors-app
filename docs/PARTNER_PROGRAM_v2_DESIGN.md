## 1️⃣ スプレッドシート構成（多言語対応）

### 設計方針
- **(a) リンク2つ ✗** → 言語別に独立した運営はメンテ負荷が高い
- **(b) タブ3つ + 統合タブ ✓** ← 採用
- **(c) サマリーダッシュボード ✓**

### スプレッドシート構成

```
📊 v7_運営管理（既存スプレッドシート）
├── パートナー（統合タブ）          ← 全申込が集約される。既存
├── パートナー_JP                  ← 日本語フォーム回答用（QUERY ビュー）
├── パートナー_EN_KM               ← 英語＋クメール語フォーム回答用（QUERY ビュー）
├── パートナーサマリー              ← 🆕 ダッシュボード（KPI 一目）
├── 紹介履歴                       ← 既存
└── 紹介予定                        ← 🆕 事前登録された紹介案件
```

### サマリーダッシュボードに表示する KPI

```
┌─────────────────────────────────────────┐
│  パートナー数      :  12 名（承認済み）   │
│  今月の新規申請   :  3 名                │
│  今月のコミッション :  $245.50          │
│  今月の支払予定   :  4 名                │
│  累計売上貢献     :  $4,820              │
├─────────────────────────────────────────┤
│  上位5名（コミッション順）              │
│  1. ヒロ      $120.00                    │
│  2. タナカ    $85.50                     │
│  ...                                     │
├─────────────────────────────────────────┤
│  国籍別分布                              │
│  日本: 8 / カンボジア: 3 / 中国: 1     │
└─────────────────────────────────────────┘
```

### 多言語フォームの実装

3つの Google Form を `createXxxApplicationForm()` 関数で自動生成：
- `createPartnerApplicationForm()` — JP（既存）
- `createPartnerApplicationFormEN()` — 🆕 英語版
- `createPartnerApplicationFormKM()` — 🆕 クメール語版

**全フォームが同じ「パートナー」シートに書き込む**ため、後続処理は完全共通化。
言語別タブは QUERY 関数で自動フィルタリング。

## 2️⃣ 紹介プロセスの設計

### 現状の課題
- 「いつ誰を紹介したか」がコード入力時まで分からない
- パートナーから個別連絡があってもシステムに残らない
- 紹介見込みリストが管理できず、フォローアップ困難

### v2 設計：事前登録 + 事後マッチング

```
┌─────────────────────────────────────────────────┐
│ パートナー（ヒロ）                              │
│   ↓ 「この人を紹介したい」                      │
│ 紹介ボット or ミニアプリ                        │
│   ↓ 紹介相手の情報を入力                        │
│ 「紹介予定」シートに記録                        │
│   ↓ 同時に                                     │
│ Admin の「紹介」トピックに通知                  │
│ ────────────────────────────────────────────  │
│ 数日後...                                      │
│ お客さま（タナカさん）が予約                    │
│   ↓ 紹介コード SM-HIRO を入力                   │
│ 「紹介履歴」に記録                              │
│   ↓ 自動マッチング                             │
│ 「紹介予定」の該当案件を「成約」に更新          │
│   ↓ ヒロさんに通知                             │
│ 「ご紹介ありがとうございました！」              │
└─────────────────────────────────────────────────┘
```

### 紹介ボット vs ミニアプリの比較

| 項目 | 紹介ボット | ミニアプリ |
|---|---|---|
| UX | 会話形式、ステップごと | フォーム形式、一括入力 |
| 入力項目数 | 多いと億劫 | 多くても OK |
| 開発工数 | 中（4-6h） | やや大（6-8h） |
| パートナー導入 | Telegram起動で即使える | 起動ボタンが必要 |
| **推奨** | ❌ | ✅（情報量が多いため） |

### 紹介専用 Bot の必要性

**結論: 専用Botは不要。既存の業務Bot（Internal Bot）を流用してミニアプリ起動するのが最適。**

理由：
- 予約Botと完全分離するのは賛成（顧客接点と混同してはいけない）
- ただし「紹介専用Bot」を新設すると、**パートナーは2つのBot を友だち追加することになる**
- 業務Bot は元々パートナー（≠ 顧客）向け。承認後のWelcome DMもここから来る
- 業務Bot から `[+ 新規紹介]` ボタンで referral.html を起動するのが UX として自然

### 紹介事前登録ミニアプリ（referral.html）の入力項目

```
🤝 新規紹介の登録

【紹介相手の情報】
■ 氏名 *               [____________]
■ 連絡先（電話/Telegram）* [____________]
■ 国籍                 [▼ 選択____]
■ 関係性               [▼ 友人/取引先/同僚/家族/その他]
■ 車種・台数           [____________]

【サービス情報】
■ 関心のあるプラン     [▼ 単発/月額/法人]
■ 希望時期             [▼ すぐ/1週間以内/今月中/未定]

【補足】
■ おすすめポイント・連絡時の注意点
   [_____________________________]
   [_____________________________]

[紹介を登録する]
```

## 4️⃣ インセンティブ設計（マルチレベル）

### 現状（v1）
- 直接紹介の売上 × 30%（一律）
- パートナー獲得ボーナスなし
- 実績による昇格なし

### v2 マルチレベル設計

#### 【Level 1】 直接紹介ボーナス（既存）
- お客さまの**初回利用売上 × 30%**
- 翌月10日 ABA 送金

#### 【Level 2】 パートナー獲得ボーナス（新規）
**「自分が紹介したパートナーが稼いだコミッションの一部をもらえる」**

- 自分が紹介したパートナーがコミッションを得たら、その**5%** が自分にも入る
- 期間限定: **被紹介パートナーの承認後12ヶ月間のみ**
- これにより「良いパートナーをリクルートする」インセンティブが生まれる

#### 【Level 3】 実績ティア（新規）
3ヶ月間の累計コミッション額に応じて昇格：

| ティア | 条件 | コミッション率 | 特典 |
|---|---|---|---|
| 🥉 ブロンズ | 累計 $0〜$200 | 30% | 通常 |
| 🥈 シルバー | 累計 $200〜$500 | **35%** | 月1回無料洗車券1枚 |
| 🥇 ゴールド | 累計 $500〜$1,500 | **40%** | 月2回無料洗車券、Welcome Kit優先発送 |
| 💎 プラチナ | 累計 $1,500以上 | **45%** | 専任窓口、特別ボーナス、年次表彰 |

#### 【Level 4】 年間ボーナス（新規）
- 年末に**累計売上TOP3** に特別ボーナス（$200/$100/$50）

### 実例（試算）

```
ヒロさんが3名のお客さまを紹介し、各 $50 の洗車を発注：
  Level 1 直接紹介: $50 × 3名 × 30% = $45.00

ヒロさんが紹介したパートナー2名が活動：
  パートナーA: 月 $30 のコミッション獲得
  パートナーB: 月 $20 のコミッション獲得
  Level 2 獲得ボーナス: ($30 + $20) × 5% = $2.50

合計: $47.50/月（v1 なら $45.00 だった）
```

### マルチレベル設計の注意点（自己制限）

⚠️ **Cambodia Pyramid Scheme 規制への配慮**:
- マルチレベルは **2階層まで** に厳格に制限
- 「他人を勧誘するだけで稼げる」構造にしない（必ず実売上を伴う）
- ボーナス率は控えめに（5%）、メインは依然として直接紹介
- 規約に明示し、年次レビューで健全性を確認

## 7️⃣ 今夜のヒロさん登録について

**結論: 既存の v1（日本語フォーム）でそのまま登録 → OK**

理由：
- v2 の多言語化はヒロさんには不要（日本人）
- v1 で登録しても v2 の対象者になる（後方互換性あり）
- 紹介事前登録機能ができるまで、ヒロさんからの個別紹介は **手動で 紹介履歴シートに記入** で対応可能

### 今夜の流れ（推奨）

1. ヒロさんに公開URL（forms.gle/xxxxx）を共有
2. その場で記入してもらう（5分）
3. Telegram の承認ボタンで「✅ 承認」
4. ヒロさんに `@SamuraiMotorsInternalBot` を友だち追加してもらい `/start` 送信
5. 受信した chat_id をパートナーシートに記入
6. `sendPartnerWelcomeDm_('PTR-260423-001')` を実行 → Welcome Kit 送信
7. 紹介コード `SM-HIRO`（または希望コード）を口頭で伝達
8. v2 設計書（本ドキュメント）を見せて「来月このシステムをリリースする」と伝える

## 8️⃣ オープン質問（要決定）

### Q1. インセンティブのコミッション率
**v1 では 30% だが、v2 マルチレベルでは Level 1 を 25% に下げて Level 2 を 5% にする案もあり得る。**
- 案A: Level 1 = 30%（現状維持）+ Level 2 = 5%（追加）→ 報酬総額は増えるがビジネスマージン圧迫
- 案B: Level 1 = 25% + Level 2 = 5%（合計 30% を維持）→ ビジネス的に持続可能

**推奨**: 案A でスタート、3-6ヶ月後に持続性を見て調整

### Q2. ティア累計期間
- 累計を**過去3ヶ月**で計算する → 直近の活動を評価
- **過去12ヶ月** で計算する → 安定性を評価
- **永続累計** で計算する → 長期パートナーを優遇

**推奨**: 過去3ヶ月（直近の活動を評価しやすい）

### Q3. 紹介事前登録の有効期限
紹介予定として登録された案件はいつまで有効？
- 30日 / 60日 / 90日 / 無期限

**推奨**: 60日（過ぎたら自動で「失効」ステータス）

### Q4. クメール語翻訳の品質保証
- AI 翻訳をベースにする → コスト 0 / 品質ばらつきあり
- カンボジア人スタッフ（ロン）にレビュー依頼 → 確実 / レビュー時間1-2h必要

**推奨**: AI ベース → ロンさんにレビュー依頼

### Q5. ヒロさんの希望コード
ヒロさんに「希望コードはありますか？」と聞いて決めておくと良い。
- HIRO / HIRO2026 / HIRO_PHNOMPENH 等
